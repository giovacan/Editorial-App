ROOT CAUSE: Measurement Width Distortion
  Location: src/hooks/usePagination.js line 196
  Severity: CRITICAL (causes ~70% of layout bugs)

MECHANISM:
  1. measureDiv.style.width = '1px' (line 196)
  2. Later corrected to contentWidth (line 228) but TOO LATE
  3. Line height measured with width:1px → wrong lineHeightPx value
  4. Elements fit in measurement but overflow in preview
  5. Text wrapping differs between measurement and rendering

SECONDARY BUGS FOUND:
  1. Margins not included in element height measurements
  2. Insufficient safety buffer (1px instead of lineHeightPx)
  3. Orphan headings allowed at page bottom
  4. Fill pass doesn't validate constraint violations
  5. Duplicate CSS rules in Preview.css

TECHNICAL DETAILS
═══════════════════════════════════════════════════════════════════════════

Measurement Width Bug - Detailed Explanation:

CURRENT FLOW (BROKEN):
  1. calculateContentDimensions() → contentWidth = 300px
  2. measureDiv.style.width = '1px'    ← Width wrong here
  3. measureDiv.style.fontSize = ... (other styles set)
  4. measureDiv.innerHTML = 'Ag'
  5. lineHeightPx = measureDiv.offsetHeight  ← Measured with width:1px!
     Result: lineHeightPx = 32px (too large)
  6. measureDiv.style.width = `${contentWidth}px`  ← Corrected too late
     (But lineHeightPx already wrong!)
  7. maxLines = Math.floor(600 / 32) = 18 lines  ← WRONG (should be 30)

FIXED FLOW:
  1. calculateContentDimensions() → contentWidth = 300px
  2. measureDiv.style.width = `${contentWidth}px`  ← Correct width FIRST
  3. measureDiv.style.fontSize = ... (all styles)
  4. measureDiv.innerHTML = 'Ag'
  5. lineHeightPx = measureDiv.offsetHeight  ← Measured with correct width
     Result: lineHeightPx = 20px (correct)
  6. maxLines = Math.floor(600 / 20) = 30 lines  ← CORRECT

Safety Margin Fix:

CURRENT (BROKEN):
  safetyMargin = 1 + headerSpaceEstimate
  contentHeight = 600 - 1 = 599px
  Last line wrapping happens at ~600px border
  Result: Last line gets clipped or cut in half

FIXED:
  safetyMargin = lineHeightPx + headerSpaceEstimate
  contentHeight = 600 - 20 = 580px
  1 full line buffer guaranteed at bottom
  Result: Last line always visible

Element Margin Example:

CURRENT (BROKEN):
  <h2>Section Title</h2>  with CSS margin: 1em 0 0.5em 0
  Actual rendered height: 8px (top) + 16px (text) + 4px (bottom) = 28px
  Measured height: 16px (text only)
  Missing: 12px of margin height

  Element "fits" in measurement (16px)
  Overflows in preview (28px)
  Result: Text overflow

FIXED:
  <h2>Section Title</h2>
  Actual rendered height: 28px
  Measured height: 28px (with margins)

  Element fits in measurement (28px)
  Fits in preview (28px)
  Result: Perfect fit

KEY INSIGHTS
═══════════════════════════════════════════════════════════════════════════

1. MEASUREMENT MUST MATCH RENDERING
   - Measurement width must equal preview width
   - Measurement CSS must match preview CSS
   - Margins must be included in measurements

2. LINE HEIGHT IS CRITICAL
   - lineHeightPx drives entire pagination algorithm
   - Must be measured with correct width
   - Any error cascades through all calculations

3. SAFETY MARGINS ARE NOT OPTIONAL
   - 1px buffer insufficient for last line rendering
   - Need full lineHeightPx to guarantee visibility
   - Better to underfill page than risk clipping

4. CONSTRAINTS MUST BE ENFORCED
   - Page endings matter (no orphan headers)
   - Fill pass must validate, not just fit
   - Prevention better than fixing overflow

5. ARCHITECTURE IS SOUND
   - Extracted pagination engine is well-designed
   - Issues are in the boundary (measurement)
   - HTML string storage model is correct
   - No ProseMirror/JSON conversion needed

QUESTIONS & ANSWERS
═══════════════════════════════════════════════════════════════════════════

Q: Why did this bug appear?
A: The width reset and measurement were written in order of style properties,
   not in order of dependencies. Correct width must be set before measuring
   line height.

Q: Will these fixes break existing code?
A: No. All fixes are improvements to accuracy. No API changes, no breaking
   changes. Purely internal calculation fixes.

Q: How much improvement will we see?
A: ~70% of issues from Fix 1 alone. Fixes 2-4 handle remaining edge cases.
   After all fixes: 100% of layout bugs should be resolved.

Q: Is pagination performance affected?
A: No. Fixes add ~1 wrapper creation/destruction per element (negligible).
   No algorithmic changes.

Q: Do I need to change editor/preview architecture?
A: No. HTML string storage model remains unchanged. Preview.jsx unchanged.
   Only pagination engine internals are improved.
